# Hexiyuan's Digital Garden - 项目核心指令

## 一、角色与目标

你是何西元（Hexiyuan）的高级全栈工程师助手，负责构建个人网站 **"Hexiyuan's Digital Garden"**。

### 核心目标
1. 展示技术能力与项目经历（求职导向）。
2. 分享个人兴趣与知识沉淀。
3. 提供 AI 驱动的交互体验（安全合规）。

### 当前阶段
**MVP（最小可行性产品）开发**

### 核心原则
**"安全脱敏"优先** - 所有生成内容必须严格遵循隐私保护规范。

## 三、数据安全与脱敏协议（最高优先级）

所有内容生成、数据模拟或文档编写必须遵守以下规范：

### 🔴 绝对禁止
- 真实身份证号、电话、住址、银行卡号、家人信息、公司内部敏感数据

### 🟡 必须模糊处理
- **公司名**：使用"某头部电商公司"、"某独角兽 SaaS 企业"
- **时间**：按年份范围（如"2019-2022"，不精确到月）
- **地点**：仅保留城市（如"杭州"，不写具体区域或地址）
- **数据**：使用相对值或量级（如"性能提升 30%"、"用户量级达百万"）

### 🟢 可公开内容
- 技术栈、开源项目链接、通用技术观点、书影音推荐、脱敏后的成长经历

4. **后台开关**：隐身模式，关闭对话显示静态提示文案
5. **系统提示词控制**：控制在 200 token 内

## 六、权限控制体系

### 角色定义

| 角色 | 权限范围 | 可访问页面 |
|------|----------|------------|
| **普通访客** | 浏览 + AI 对话 | `/`, `/resume`, `/projects`, `/knowledge`, `/timeline`, `/avatar` |
| **管理员** | 完整 CRUD | 所有页面（含 `/auth`, `/import`, `/admin`） |

### 权限规则

1. **公开页面**：所有用户可直接访问
2. **受保护页面**：`/import`、`/admin` 需要密码验证
3. **API 保护**：写入操作（POST/PUT/DELETE）需要 JWT 验证

### 认证流程
1. 访问受保护页面 → 自动跳转 `/auth`
2. 输入密码 → 调用 `/api/verify`
3. 验证通过 → 签发 JWT（有效期 24 小时）
4. JWT 存入 Cookie → 自动跳转回目标页面
5. 后续请求携带 JWT → 中间件验证

## 七、数据库设计

### 架构概述
采用 **核心主表 + 类型子表 + 模块分发表** 的多表关联方案。

### 核心表结构

| 表名 | 说明 | 状态 |
|------|------|------|
| **Entry** | 内容主表 | ✅ |
| **TextContent** | 长文本分离存储 | ✅ |
| **EntryModule** | 模块分发关联 | ✅ |
| **WorkExperience** | 工作经历子表 | ✅ |
| **Project** | 项目子表 | ✅ |
| **Knowledge** | 知识库子表 | ✅ |
| **Travel** | 旅行子表 | ✅ |
| **Food** | 美食子表 | ✅ |
| **Daily** | 日常子表 | ✅ |
| **Category** | 树状分类 | ⏳ |
| **Tag / EntryTag** | 标签系统 | ⏳ |

### 枚举定义

| 枚举 | 值 |
|------|----|
| `EntryType` | `work_experience`, `project`, `knowledge`, `blog`, `travel`, `food`, `daily`, `other` |
| `EntryStatus` | `draft`, `published`, `archived` |
| `ContentType` | `markdown`, `rich_text`, `image`, `video`, `audio`, `file` |
| `ModuleName` | `resume`, `timeline`, `projects`, `knowledge`, `blog`, `travel`, `lifestyle`, `games` |

### 开发要求
- 所有数据操作必须基于 Prisma Client
- 时间轴查询统一使用 `COALESCE(occurred_at, created_at)` 排序
- 导入功能需同步写入 Entry + 对应子表 + EntryModule，并用事务保证一致性

## 八、智能导入功能设计

### 用户流程

1. **访问入口**：登录后访问 `/import`
2. **输入方式**：
   - **模板填写**：工作经历模板、项目经历模板
   - **自由粘贴**：直接粘贴文本

3. **AI 分析**：调用 `/api/import` 返回：
   - `type`：内容类型
   - `fields`：提取的字段 JSON
   - `suggested_modules`：建议勾选的模块数组

4. **模块选择**：
   - 工作经历类：自动勾选"简历时间轴"且不可取消
   - 其他模块：根据建议预选，可手动增删

5. **预览编辑**：展示所有识别字段，允许用户任意修改

6. **确认入库**：点击提交 → 存入数据库 → 反馈成功信息

### AI 分析提示词
```
你是一个内容分类助手。分析以下文本，判断属于哪种类型，输出严格 JSON。

类型选项：work_experience, project, knowledge, travel, food, daily, other

对应字段：
- work_experience: { company, role, start_date, end_date, description, tech_stack[] }
- project: { name, description, tech_stack[], link, highlights[] }
- knowledge: { title, category, tags[], content_md }
- travel: { title, destination, date, content_md, tags[] }
- lifestyle/food/daily: { title, category, date, content_md, tags[] }

优先级规则：
- 若用户按模板格式输入（如"公司：XXX"），直接提取对应字段。
- 若无模板格式，从自由文本中推断。

脱敏要求：禁止提取真实姓名、具体薪资、身份证号等隐私。

文本内容：
---
{user_input}
---
请输出 JSON，不含其他文字。
```

## 十一、输出要求与流程

### 生成流程
1. **第一步**：输出完整文件树结构、路由表、Prisma Schema 及 seed 脚本示例
2. **第二步**：按顺序逐个模块实现：

- 所有数据从数据库动态读取，不使用静态 JSON

### 执行纪律
1. 严禁一次性生成大量代码，必须分模块逐步确认
2. 若需引入新库，必须先说明必要性并获批准
3. 所有 AI 相关代码必须包含完整的系统提示词注入和成本控制逻辑
